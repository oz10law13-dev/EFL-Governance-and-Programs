# EFL Kernel — Claude Working Rules

## 1. Always show a diff summary before making any edits.

Before touching any file, describe exactly what will change: which file, which lines, and why. Do not make the edit until that summary has been stated.

## 2. No silent refactors — every change must be explained before it is made.

Every modification — including whitespace, import reordering, or comment updates — must be described and justified before the edit tool is called. "Cleaning up" or "while I'm here" changes are not permitted.

## 3. Frozen spec files are immutable.

Files matching `efl_kernel/specs/*.json` must never be edited in place. If a spec change is required:
1. Create a new versioned frozen file (e.g. `EFL_RAL_v1_3_0_frozen.json`).
2. Recompute `documentHash` and any affected `registryHash` values using `canonicalize_and_hash` from `efl_kernel/kernel/ral.py`.
3. Update all `moduleRegistration` entries in the new spec to reflect the new versions and hashes.
4. Update the import path in `efl_kernel/kernel/ral.py` (`SPEC_PATH`) to point to the new file.

## 4. After every change: run tests, then commit.

```
cd C:\EFL-Kernel
python -m pytest -q
```

All tests must pass before committing. Commit message format:

```
<phase>: <what changed>
```

Examples: `conformance: fix step 9 cap breach logic`, `spec: add RAL.NEWCODE to v1.3.0 frozen spec`

## 5. Never source aggregates from the input payload.

Load totals, prior session timestamps, and season phase counts must always be fetched from the dependency provider (`KernelDependencyProvider`). These values must never be read from `raw_input` or any caller-supplied dict. This applies to all gate implementations and any future kernel steps.

## 6. KDOValidator allowed sets must be sourced from the frozen RAL spec at runtime.

Allowed values for fields such as `finalEffectiveLabel`, `finalSeverity`, and `finalPublishState` must be derived from `RAL_SPEC` at import time (e.g. from `RALPrecedenceRule`, `RALPublishStateDerivation`). They must not be hardcoded as Python literals in `kdo.py` or any validator.

## 7. Phase execution standard — apply to every phase prompt.

Every phase prompt and every phase execution must follow this structure
in order. Deviating from this structure is not permitted.

### STEP 0 — Read before touching anything

Before writing any code or making any edit, run:

    cat CLAUDE.md

Then read every file listed in the prompt's read list using cat. Then
confirm and print every item in the confirm list. Do not proceed until
all items are confirmed and printed in the terminal.

### Minimum confirm-and-print items for every phase

1. Full suite baseline: `python -m pytest -q 2>&1 | tail -5`
2. Git status: `git status` — must be clean working tree
3. Git HEAD: `git log --oneline -3`
4. All signatures, return types, line numbers, or fixture shapes
   specified in the prompt's confirm list

Do not make any changes until all items are confirmed and printed.

### Verify before committing

Run in this order:

    python -m pytest efl_kernel/tests/test_phase<N>.py -v 2>&1 | tail -15
    python -m pytest -q 2>&1 | tail -5
    git diff --name-only | grep -v test_phase<N> | grep test_ || echo "no existing tests modified"

Do not commit until all three pass cleanly.

### Final check — every phase, before push

    git status
    git log --oneline -5
    git log --oneline origin/main -5

Local HEAD must equal origin/main HEAD.

### Commit 2 always covers spec doc + roadmap — two locations mandatory

Every phase closes with a BINDING spec doc and a roadmap update.
The roadmap must be updated in both locations — missing either is an
error:
  1. Header block — suite count, last completed phase, commit hash
  2. Tier A table — phase row struck through, next phase marked NEXT

## 8. LLM boundary rule — applies to Phase 18 and all authoring phases.

The LLM may propose. The gate layer decides. These roles are never
reversed.

LLM output is NEVER written directly to:
- Any KDO field
- Any finalPublishState value
- Any violation code or reason code
- Any frozen spec file or field
- Any audit store record (kdo_log, override_ledger)

The only permitted LLM output path is:
  LLM generates a candidate payload
  → candidate is submitted to the gate layer via the normal evaluate path
  → the gate layer produces the KDO
  → the KDO determines publish state

The system must produce identical legality outcomes whether the
candidate payload was generated by an LLM, a human, or a test fixture.
LLM involvement is invisible to the gate layer. The gate does not know
and must not care how a payload was produced.

In Phase 18, no LLM is wired. The proposal engine is purely
deterministic. This rule records the boundary so that when LLM
integration is added in a future phase, the boundary is already
encoded and cannot be crossed.

## 9. No new routes without tests.

Every new HTTP route must have at minimum:
  - One test for the 200/201 happy path
  - One test for the expected rejection/error path (4xx)

A route committed without tests is not considered complete.
Do not commit a route-only diff — route and tests go in the same
commit unless the prompt explicitly stages them separately.
